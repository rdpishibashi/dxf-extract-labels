# TERMINAL_DETECTION.md — 端子一覧抽出（「端子一覧を抽出」オプション）

> 全体構成・アーキテクチャは [OVERVIEW.md](OVERVIEW.md) を参照。

## 概要

タイトルが「UNIT内結線図」の図面を対象に、`TB`で始まる端子台ラベル（例:
`TB001`、`TBN241 (24A)`）に対応する端子台矩形を検出し、矩形内に記載された
端子番号（数字のみのラベル）を抽出し、`TB List` シートに出力する機能。
`utils/terminal_detector.py` の `analyze_dxf_terminals()` /
`build_terminal_rows()`、`app.py` の「端子一覧を抽出」チェックボックスで
実装。

`region_detector.py`（矩形領域抽出）とは独立したモジュールだが、矩形の辺の
識別キー（`lineweight=25`/`color=2`）は共有している。両者を区別するため、
端子台矩形は**必ずCIRCLEによる橋渡しを1箇所以上持つ**ことを要件とする
（後述）。

## 候補ラベルの判定

正規表現 `^TB[A-Z0-9]+`（NFKC正規化後の文字列に対し先頭一致）で判定する。
「TB」で始まり、その直後に英大文字・数字が1文字以上続くもの（後続に続く
文字列＝スペース・カッコ・電流値等は問わない）。

この条件により、以下を構造的に候補から除外する（ユーザー確認済み）:

- `TB取付板`（TBの直後が漢字）— 部品〈端子台〉の取付板を指す注記であり、
  端子台そのものではない
- `TB COMPONENT`（TBの直後がスペース）— サブタイトルとして現れる文字列
  だが、ラベルとしても候補から除外される

以前は「対応する矩形が見つからない」ことをもって除外していたが（マッチング
段階のフォールバック）、この判定により候補選定の時点で明示的に除外される。

## 対象ファイル判定

1. `extract_labels()` でタイトル・サブタイトル・図番を抽出する。
2. タイトル（NFKC正規化後）に「UNIT内結線図」を含まない図面は対象外
   （`is_target=False`、以降の矩形検出は行わない）。
3. サブタイトルが（NFKC正規化・前後空白除去後）「TB COMPONENT」と完全一致
   する図面は対象外。このサブタイトルの図面は端子台の**全端子番号一覧**
   ページであり、そこに記載される数字は「使用中の結線」ではなく「端子台が
   物理的に持つ全端子番号」を意味するため、他の図面と同列に扱えない
   （ユーザー確認済み、`EE6892-039-82A.dxf` で確認）。

## 端子台矩形の識別キー

- **矩形の辺**: `lineweight=25` かつ `color=2`(ACI黄) の LINE
  （矩形領域抽出＝`region_detector.py` の領域境界線と同じ識別キー）
- **端子（ギャップ補完）**: `lineweight=50` かつ `color=4`(ACI水色) の CIRCLE。
  部品（端子記号）が辺の途中に配置されて LINE を途切れさせている箇所を、
  この CIRCLE が橋渡しする。

**CIRCLE による橋渡しが1箇所以上ある矩形のみを端子台矩形として扱う。**
LINEのみで閉じた矩形は、矩形領域抽出の領域境界線そのものである可能性があり
（同じ `lineweight=25`/`color=2` の識別キーを使うため）判別できない。この
制約により、CIRCLE橋渡しを持たない矩形（領域境界線由来の大きな矩形等）を
安全に除外できる（ユーザー確認済み）。

## ジオメトリ収集の対象範囲（`_collect_geometry`、v1.8.2で確定）

Model Space に何らかの内容（texts・lines・circles のいずれか）があれば
常に Model Space のみを対象にする。Model Space が**完全に空の場合のみ**、
Model 以外のレイアウト（Paper Space等）を順に試し、内容が見つかった
レイアウト自身の texts/lines/circles を使う（レイアウトをまたいで
混在させない）。

図面枠・端子台矩形一式が Model Space ではなく Paper Space レイアウトに
配置されている図面（`EE5322-455-01B.dxf`等）が実データに存在するため、
Model 以外のレイアウトを走査する対応自体は必要（`region_detector.py`・
`ref_designator.py`で確認・修正済みの不具合と同根）。ただし
**Model Space と Paper Space は独立した座標系のため、一方のレイアウトの
ラベルと別レイアウトの矩形辺LINEを組み合わせると、TBラベルと矩形の距離
ベースの対応判定が座標的に成立しなくなる**（v1.8.0では無条件に両レイアウト
の図形をまとめて収集していたが、`ref_designator.py`側で`EE6892-455B.dxf`の
`CB001`等が誤って除外される不具合が発生したため、v1.8.2で
「同一レイアウト内で完結させる」方針に修正）。実サンプルの
「UNIT内結線図」ではこの経路はまだ発生していない（潜在的な不具合として
予防的に修正、合成DXFの単体テストで検証）。

## 検出アルゴリズム（`_build_rect_candidates`）

1. LINE(`lineweight=25`/`color=2`)の全端点を収集し、Union-Findで連結成分に
   まとめる。同一LINEの2端点、および座標が一致する（許容誤差0.15）端点
   同士をそれぞれ結合する。
2. CIRCLE(`lineweight=50`/`color=4`)ごとに、円中心と同一x(縦方向)/同一y
   (横方向)にある最近傍の端点対（円を挟んで両側にあるもの）を見つけて結合
   する（橋渡し）。橋渡しに使われた端点は記録しておき、後段で「この連結
   成分はCIRCLE橋渡しを持つか」を判定する。
3. 連結成分ごとに、CIRCLE橋渡しを持たない成分を除外したうえで、成分に属する
   全端点のbounding boxを矩形候補とする。bbox の4辺それぞれについて、その
   辺に沿うLINE（両端点がその辺の座標に一致し、辺方向に一定の長さを持つ
   もの）が成分内に少なくとも1本存在することを確認し、4辺すべて確認できた
   ものだけを矩形候補として採用する。

## ラベル-矩形の対応判定（`_match_labels_to_rects`）

マージン（距離のしきい値）をオプションとして公開せず、常に矩形を構造的に
探索して対応付ける（ユーザー指示。当初案の「近傍マージンをオプション化
〈既定0.1〉」は、実データでラベル挿入点から矩形の辺までの距離が1.25〜4.1
あり、0.1では検出できないことが判明したため撤回した）。

ラベルは矩形の四辺のうちいずれかの辺の近傍にあり、実データでは「矩形の
直上」（最頻出）と「矩形の直下」の両方が観測されている
（`EE6892-039-05B.dxf` の `TBLS (17.5A)` は矩形の下）。

**隣接矩形への上下衝突**: 同一の物理位置に複数の矩形が隣接して存在する
場合、1つのラベルが「上の矩形の下辺」にも「下の矩形の上辺」にも近い
距離を持つことがある。`EE6492-039-38A.dxf` で実際に発生: ラベル
`TBP044 (17.5A)`（y=254.0）は、下の矩形（y=245.0-250.0、primary距離4.0）
と上の矩形（y=257.5-262.5、secondary距離3.5）の中間に位置し、単純な最短
距離だけで判定すると secondary距離がより小さい上の矩形を誤って選んで
しまう。一方 `TBN241 (24A)`（y=266.5）は上の矩形へのprimary距離4.0
（唯一の候補）しか持たない。

対策として「ラベルが矩形の上（primary）」を「ラベルが矩形の下
（secondary）」より**常に優先**し、全ラベル・全矩形の候補ペアを
`(0 if primary else 1, 距離)` 昇順でソートしたうえで、貪欲に1:1マッチング
する（`_match_labels_to_rects`）。上記の例では
`(0, 4.0, TBN241, 上矩形)` と `(0, 4.0, TBP044, 下矩形)` が
`(1, 3.5, TBP044, 上矩形)` より先に処理されるため、TBP044は正しく下の
矩形に、TBN241は上の矩形に割り当てられる。

対応する矩形が見つからない（距離が`_MATCH_MAX_DIST`(10.0)を超える、または
他のラベルに矩形を先取りされた）TBラベルは `unmatched_labels` に入り、
出力（`entries`）には含まれない。前述の候補ラベル判定で「TB取付板」等の
非端子台ラベルは既に除外されているため、実データでこの状態になるのは
候補パターンには一致するが物理的に対応する矩形を持たない、より稀な
ケースに限られる。

### 90°回転図面への対応

図面全体が90°回転している場合、「上/下」の優先関係が「右端/左端」に
入れ替わる。回転検出は `region_detector._label_rotation_angle` と同じ
ロジックでラベルの回転角を求め、過半数が+90°付近なら`rotated_mode='right'`
（ラベルが矩形の右にあるほど優先）、-90°付近なら`'left'`とする
（`region_detector._rotated_edge_roles` と同じ規約: 回転角+90°多数派なら
primary=右端、-90°多数派ならprimary=左端）。

**実データに90°回転した「UNIT内結線図」のサンプルが無いため、この分岐は
合成DXFの単体テストのみで検証している。** 実サンプルが見つかった場合は
方向の妥当性を再確認すること。

## 端子番号の抽出（`_collect_digits`）

マッチした矩形のbounding box（パディング0.5）内にあるラベルのうち、
NFKC正規化後に数字のみ（`str.isdigit()`）のものを整数として収集する
（全角数字「１」「２」等も半角相当で判定・抽出する）。

## 端子台ラベルの集約（ファイル内・`analyze_dxf_terminals`）

同一ラベルテキスト（NFKC正規化・前後空白除去）が1ファイル内の複数の矩形
にまたがる場合（例: `EE6892-039-05B.dxf` の `TBLS (17.5A)` は2つの矩形
`[1, 2]`・`[3]` にまたがる）、端子番号を1つのリストに集約する（`grouped`
dict）。**この段階では重複除去しない**（同じ番号が複数の矩形にまたがって
実際に複数回現れた場合、その情報を保持しておき、後段の `build_terminal_rows()`
で出現回数として表示するため）。ファイルをまたいだ集約はこの関数では行わない
（`entries` は常に単一ファイル分）。

## TB List シートの集計（ファイル横断・`build_terminal_rows`）

「端子台」（ラベルの正規化済みテキスト）で**ファイルをまたいでユニーク**を
とる。同じ端子台名が複数ファイルにまたがって登場する場合、それらの端子番号
・図番を1行に統合する。

- **端子No.**: 全ファイル・全矩形から集めた番号を `collections.Counter` で
  集計し、出現回数が2回以上の値は `N(件数)` の形式で表示する（例:
  `[1, 2, 3, 7, 7]` → `1, 2, 3, 7(2)`）。1回のみ出現する番号はそのまま
  表示する（`_format_numbers_with_counts`）。
- **図番**: 該当ファイルの図番（未抽出時はファイル名〔拡張子なし〕に
  フォールバック）を集合として集め、重複を除いてABC順（`sorted()`）に
  カンマ区切りで列挙する。
- **行順**: 「端子台」のABC順（`sorted()`）。

### 矩形が見つからない候補ラベル（`端子検出不可`行）

候補パターン（`^TB[A-Z0-9]+`）には一致したが、対応する矩形が見つからな
かったラベル（`analyze_dxf_terminals()` の `unmatched_labels`）は、通常の
集計行の**末尾に1行の空行を挟んで**追加する（ユーザー指定の仕様）。

| 列 | 内容 |
|----|------|
| `端子台` | 固定文字列 `端子検出不可`（`_UNMATCHED_LABEL_MARKER`） |
| `端子No.` | 矩形が見つからなかったラベルのテキスト |
| `図番` | そのラベルが検出されたファイルの図番（複数ファイルにまたがる場合はABC順でカンマ区切り）|

通常行と同様に、同じラベルが複数ファイルにまたがる場合は図番を統合し、
行自体はラベルのABC順に並べる。空行は `unmatched_labels` を持つファイルが
1件も無い場合は追加しない。実データにこの状態を再現するサンプルが無いため
（`TB取付板`等は候補ラベル判定の段階で既に除外される）、単体テスト
（合成データ）のみで検証している。

## Excel 出力（`build_terminal_rows` / `excel_output._write_terminal_sheet`）

| シート | 内容 |
|--------|------|
| `TB List` | `端子台` / `端子No.`（カンマ区切り・件数付き・昇順） / `図番`（カンマ区切り・ABC順）。末尾に矩形未検出ラベルの `端子検出不可` ブロックを含むことがある |

- シート位置: 既定モード・「以外も抽出」ONいずれも `Total` シート直後。
  領域モード（`Total`シートが無い）では `領域別ラベル一覧` シート直後。
- 対象外ファイル（`is_target=False`）は集計対象に含めない。
- `terminal_results` が `None`、または対象ファイルが1件も無い場合、
  シート自体を作らない（`_write_terminal_sheet` が `None` を返す）。

**全シートの先頭行固定**: この機能の追加に合わせて、`TB List` シートを
含む全シート（`Summary`/`Total`/各ファイルシート/`領域一覧`/
`領域別ラベル一覧`）に `freeze_panes(1, 0)` を適用した
（`excel_output._write_header_row()` に集約。`領域別ラベル一覧` のみ
ヘッダー行を手書きしているため個別に `freeze_panes` を呼んでいる）。

## UI（`app.py`）

「端子一覧を抽出」チェックボックス（オプション欄、既定OFF）。マージン等の
詳細設定は持たない（前述のとおりオプション化を撤回したため）。

処理フロー: 「ラベルを抽出」ボタン押下時に、モード（機器符号（候補）
パイプライン/「以外も抽出」・領域あり/なし）を問わず一度だけ
`_compute_terminal_results()` を実行し `session_state['terminal_results']`
に保持する。既定モード（「機器符号（候補）以外も抽出」OFF）は Excel 生成
が「選択完了」まで遅延されるため、保持しておいた `terminal_results` を
「選択完了」時の `create_region_excel_output()` /
`create_ref_designator_excel_output()` 呼び出しに渡す。「新しい抽出を
開始」・次回の「ラベルを抽出」実行時にクリアする。

## 設定・定数

矩形検出・マッチングの内部定数（UIには公開しない、`utils/terminal_detector.py`
モジュール冒頭）:

| 定数 | 既定 | 役割 |
|------|------|------|
| `TB_LINE_WEIGHT` / `TB_LINE_COLOR` | 25 / 2 | 矩形の辺のLINE識別キー |
| `TB_CIRCLE_WEIGHT` / `TB_CIRCLE_COLOR` | 50 / 4 | 橋渡しCIRCLEの識別キー |
| `_ENDPOINT_TOL` | 0.15 | 同一点とみなす座標許容誤差 |
| `_BRIDGE_AXIS_TOL` | 0.5 | 円が橋渡しする軸(x/y)の一致許容誤差 |
| `_BRIDGE_SEARCH_RANGE` | 15.0 | 円から橋渡し相手のLINE端点を探す最大距離 |
| `_MATCH_MAX_DIST` | 10.0 | ラベル-矩形間のマッチとして許容する最大距離 |
| `_MATCH_XY_TOL` | 1.0 | ラベルが矩形の幅(高さ)範囲内とみなす許容誤差 |
| `_MIN_RECT_SIZE` | 1.0 | 矩形として扱う最小の幅・高さ |
| `_ROTATION_THRESHOLD` | 0.5 | 図面回転判定の閾値（ラベルの過半数が90°回転で「回転」）|
